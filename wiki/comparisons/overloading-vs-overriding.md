# 오버로딩 vs 오버라이딩

## 비교 목적

두 용어 모두 “같은 이름”과 관련되어 헷갈리기 쉽다. 오버로딩은 같은 클래스 안에서 이름을 여러 방식으로 정의하는 것이고, 오버라이딩은 상속받은 메서드를 자식 클래스에서 다시 정의하는 것이다.

## 한눈에 보기

| 항목 | 오버로딩 | 오버라이딩 |
|---|---|---|
| 위치 | 같은 클래스 안 | 부모-자식 상속 관계 |
| 기준 | 매개변수 개수·타입·순서 차이 | 부모 메서드 시그니처와 호환 |
| 목적 | 다양한 입력 방식 제공 | 자식 클래스 상황에 맞게 동작 변경 |
| 수업 예 | 생성자 여러 개 | `toString()` 재정의, 음료별 출력 |
| 연결 개념 | 생성자, 메서드 | 상속, 다형성 |
| 선택 시점 | 컴파일 시 인수와 시그니처로 선택 | 런타임 실제 객체의 재정의 구현 선택 |
| 제약 | 반환 타입만 달라서는 성립하지 않음 | `private`, `static`, `final` 메서드는 일반 인스턴스 메서드처럼 오버라이딩되지 않음 |

## 언제 무엇을 쓰는가

1. **상품 생성 입력이 다를 때 — 오버로딩:** `Product03(String name, String inputdate)`와 `Product03(String name, int price, String inputdate)`를 함께 두어 단가를 생략한 상품과 직접 지정한 상품을 같은 생성자 이름으로 만든다.
2. **출력 방식이 객체마다 다를 때 — 오버라이딩:** `Object`에서 물려받은 `toString()`을 `Animal`이나 음료 클래스가 자신의 필드에 맞게 다시 정의한다.
3. **둘을 함께 쓰는 상황:** 여러 생성자로 객체를 만든 뒤, 부모 타입 배열에 담아 공통 메서드를 호출하면 생성 시점에는 오버로딩 선택이, 실행 시점에는 오버라이딩된 동작이 관여할 수 있다.

## 관련 수업 예시

03-10의 `Product03` 객체 배열은 서로 다른 생성자 호출로 상품을 만들었고, 03-11의 상속 실습은 `toString()`과 음료 정보 출력 메서드를 자식 상황에 맞게 재정의했다. 두 문법은 이름이 비슷하지만 적용 시점과 해결하려는 문제가 다르다.

## 헷갈리기 쉬운 포인트

- 오버로딩은 반환 타입만 다르게 해서는 성립하지 않는다.
- 오버라이딩은 부모 메서드와 이름·매개변수 구조가 맞아야 한다.
- 자식 클래스에 이름만 같은 새 메서드를 만들었다고 모두 오버라이딩은 아니다. 부모 메서드와 재정의 관계가 성립하는지 `@Override`로 확인하는 편이 안전하다.
- 다형성에서는 부모 타입으로 호출해도 실제 자식 객체의 오버라이딩 메서드가 실행될 수 있다.
